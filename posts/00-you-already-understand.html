So, What Is a Microcontroller?

You know how a web server has a CPU, some RAM, and a hard drive? A microcontroller is all of that squeezed onto a single chip the size of your fingernail. It's a complete, self-contained computer.

First Things First: What Is "High" and "Low"?

In web development, everything is data: strings, numbers, JSON objects. In embedded programming, the most basic unit of communication is voltage.

A microcontroller pin can be in one of two states: high (the chip is pushing voltage out, typically 3.3 volts) or low (the pin is connected to ground, 0 volts). That's it. Every single thing a microcontroller does in the physical world starts with making pins go high or low.

When you write code for a microcontroller, setting a pin "high" is the equivalent of element.style.display = "block" in the browser. It's the simplest possible action, and everything else builds on top of it.

GPIO: The Pins That Connect to the Real World

GPIO stands for General Purpose Input/Output. These are the physical metal pins on a microcontroller chip that you can connect wires to. Each one can be configured independently as either an input (reading a signal from the outside world) or an output (sending a signal out).

A typical microcontroller has anywhere from 10 to 40+ GPIO pins. Each one is like a tiny, individually addressable port between your software and the physical world.

Web dev analogy

GPIO pins are like the DOM API for the physical world. Just as document.getElementById() lets your JavaScript interact with elements on a page, GPIO lets your code interact with real, physical things: LEDs, buttons, motors, sensors, and displays.

Output: Making Things Happen

When a GPIO pin is configured as an output, your code controls whether it's high or low. Connect an LED (with a resistor to limit the current, otherwise you'd burn it out), and you can turn it on and off. Connect a relay, and you can switch a mains-powered appliance. Connect a motor driver, and you can spin a motor.

Wait, What Is a Resistor?

You'll notice the circuit above includes a component labelled "330Ω." That's a resistor, and it's one of the most basic electronic components you'll encounter.

A resistor does exactly what the name suggests: it resists the flow of electrical current. Think of water flowing through a pipe. The voltage is like water pressure, the current is the amount of water flowing, and a resistor is a narrower section of pipe that limits how much water can pass through.

Without the resistor in our LED circuit, too much current would flow through the LED and it would burn out almost immediately. The resistor limits the current to a safe level. The value "330Ω" (330 ohms) is chosen specifically for common LEDs running at 3.3V, and the maths behind it comes from Ohm's Law, which relates voltage, current, and resistance.

Input: Reading the Outside World

When a GPIO pin is configured as an input, the microcontroller reads whatever voltage is present on the pin and reports it to your code as a 1 (high) or 0 (low). This is how you detect a button press, read a switch position, or receive a digital signal from a sensor.

But there's a catch that trips up every beginner: what happens when an input pin has nothing connected to it?

The Floating Pin Problem

An unconnected input pin doesn't default to 0. Instead, it "floats," picking up stray electrical noise from nearby circuits, radio waves, and even the 50/60Hz hum from your power lines. The result looks like this:

The pin is floating. It reads random noise. Completely useless.

A pull-up resistor gently connects the pin to the supply voltage, so it reads as 1 by default. A pull-down resistor gently connects it to ground, so it reads as 0 by default. Most microcontrollers have these resistors built in, and you just need to enable them in your code. When a button or sensor drives the pin, it easily overpowers the weak pull resistor and changes the state.

How Your Code Actually Controls Hardware

Here's where things get really interesting, and really different from web development.

In the browser, when you write element.style.color = "red", you're calling into a rendering engine that interprets your instruction and eventually updates pixels on a screen. There are many layers between your code and the physical result.

On a microcontroller, the gap between code and hardware is almost nonexistent. The chip's hardware peripherals (like GPIO pins) are mapped to specific memory addresses. When your code writes a value to one of these addresses, the hardware responds directly. There is no operating system, no driver layer, and no abstraction in between.

Web dev analogy

Imagine if, instead of calling fetch() to make a network request, you could just write a URL into a specific memory location and the network card would immediately start sending packets. That's roughly how memory-mapped I/O works. Writing to an address is the action.

This is why embedded programming feels so direct. When you set a bit in a register, the corresponding pin changes voltage within nanoseconds. There's no event loop, no garbage collector, and no asynchronous callback. It just happens.

Polling vs Interrupts: Two Ways to Wait for Things

Let's say you have a button connected to a GPIO input pin and you want your code to respond when someone presses it. There are two fundamentally different approaches.

Polling: Checking Over and Over

The simplest approach is to check the pin state in a loop, over and over, as fast as the CPU can go. This is called polling. It works, but it's like refreshing your email inbox every second to see if you've got a new message. Technically effective. Enormously wasteful.

Interrupts: "Call Me When It Happens"

Interrupts are the embedded equivalent of event listeners. Instead of checking constantly, you tell the hardware "when this pin changes, run this function." The CPU is then free to do other things, or even go to sleep to save power. The moment the pin state changes, the hardware automatically pauses whatever the CPU was doing, runs your handler function, and then resumes.

What Can You Actually Build With This?

Everything we've covered, GPIO, inputs, outputs, interrupts, is the foundation. With just these basics, people build real products and projects:

Home automation. Reading temperature and humidity sensors, controlling relays to switch lights and fans, sending data over WiFi to a dashboard. The ESP32 has WiFi built in, so your web skills translate directly when building the interface.

Robotics. Reading encoders to know how far a wheel has turned, driving motors through H-bridge controllers, processing sensor data to navigate. Every input and output goes through GPIO.

Wearables and IoT devices. Battery-powered sensors that wake up, take a measurement, transmit data over Bluetooth, and go back to sleep. These devices run for months on a coin cell battery because interrupts let the CPU sleep 99.9% of the time.

Musical instruments and audio. Generating precise waveforms with PWM (which we'll cover in a later post), reading buttons and knobs, driving displays. Microcontrollers are inside most guitar pedals and synthesisers.
