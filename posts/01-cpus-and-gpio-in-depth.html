CPUs and GPIO in Depth

Registers, memory-mapped I/O, and what's actually happening at the hardware level when you set a pin high.

Prerequisites: This post builds on Part 00, where we covered what a microcontroller is, what GPIO pins do, and the basics of high/low, pull resistors, and interrupts. If terms like "floating pin" or "pull-up resistor" are new to you, start there.

The Microcontroller as a Self-Contained System

In Part 00 we compared a microcontroller to a laptop. Now let's look more closely at what's actually inside one. A typical microcontroller like the STM32F103 contains a 72MHz ARM Cortex-M3 CPU, 64KB of flash memory for your program, 20KB of RAM for runtime data, and dozens of hardware peripherals, all on a single chip.

To put the memory in perspective: 64KB of flash is enough to hold tens of thousands of lines of compiled C code. 20KB of RAM is enough to manage sensor readings, communication buffers, and application state for most embedded projects. You won't be loading images or parsing megabytes of JSON, but you'll have more than enough for the kind of work microcontrollers are designed to do.

The CPU clock speed matters too, but probably not the way you'd expect. At 72MHz, the CPU executes one instruction roughly every 14 nanoseconds. That's fast enough to toggle a GPIO pin millions of times per second, process incoming serial data in real time, and run control algorithms for motors and robotics. The reason microcontrollers use lower clock speeds isn't a lack of capability. It's a deliberate trade-off for lower power consumption and simpler, more predictable timing.

Memory-Mapped I/O: The Big Idea

In Part 00 we introduced the concept that writing to a memory address can control hardware. Here's how that actually works.

A microcontroller's memory space is divided into regions. Some addresses point to flash memory (where your program lives). Some point to RAM (where your variables live). And some addresses point directly to hardware peripherals. This last category is what makes embedded programming fundamentally different from application development.

On an STM32, the GPIO peripherals live at addresses starting from 0x40010800. Each GPIO port (a group of 16 pins) occupies a block of memory at a known offset. When your code writes a value to one of these addresses, the CPU puts the value onto the internal memory bus, and the GPIO hardware responds by changing the electrical state of physical pins.

Here's what that looks like in C:

// Set pin 5 of GPIO port A to HIGH
*(volatile uint32_t *)0x40010810 |= (1 << 5);

Let's break that down piece by piece. 0x40010810 is the address of the Output Data Register (ODR) for GPIO port A. The (volatile uint32_t *) tells the compiler to treat this number as a pointer to a 32-bit hardware register. The |= (1 << 5) sets bit 5 to 1 without disturbing the other bits. And volatile prevents the compiler from optimising away the write, since the compiler can't know that writing to this address has a physical side effect.

In practice, you rarely write raw addresses like this. Chip manufacturers provide header files that give these addresses human-readable names:

// Same thing, but readable
GPIOA->ODR |= (1 << 5);

Behind the scenes, GPIOA is just a pointer to the address 0x40010800, and ODR is a field at offset 0x10 within that structure. The compiler resolves it to the same memory access.

GPIO Registers: The Full Picture

In Part 00, we treated GPIO as a simple on/off switch. In reality, each GPIO port exposes a set of registers that give you fine-grained control over the pin's behaviour. Using the STM32 as an example, since most modern microcontrollers look similar, here's what you get:

RegisterWhat It Does
MODERConfigures each pin's mode: input, output, alternate function, or analog. Two bits per pin, so a 32-bit register covers all 16 pins.
OTYPERSets each output as push-pull (drives both high and low) or open-drain (drives low only, floats when high). One bit per pin.
OSPEEDRControls the output slew rate, i.e. how fast the voltage transitions between states. Faster means cleaner signals at high frequencies, but more electromagnetic noise.
PUPDREnables the internal pull-up or pull-down resistor. This is the register you configure to solve the floating pin problem from Part 00.
IDRInput Data Register. Read-only. Each bit reflects the current voltage state of the corresponding pin.
ODROutput Data Register. Read-write. Setting a bit drives the corresponding pin high; clearing it drives low.
BSRRBit Set/Reset Register. Write-only. Lets you set and clear specific pins in a single atomic operation.

Why BSRR Exists

The BSRR register solves a subtle but important problem. Suppose you want to set pin 5 high and pin 3 low. Using ODR, you'd read the current value, modify two bits, and write it back. But what if an interrupt fires between the read and the write, and the interrupt handler also modifies ODR? Your write would overwrite the interrupt handler's changes. This is called a race condition, and it creates bugs that appear intermittently and are miserable to track down.

BSRR avoids this entirely. The lower 16 bits specify which pins to set high, and the upper 16 bits specify which pins to clear. The hardware processes the entire write as a single atomic operation:

// Set pin 5 high AND pin 3 low, atomically
GPIOA->BSRR = (1 << 5) | (1 << (3 + 16));

No read-modify-write cycle means no window for an interrupt to interfere. This kind of hardware-level atomicity is something you rarely think about in application programming, but it becomes second nature in embedded work.

Alternate Functions: One Pin, Many Jobs

A chip like the STM32F103 has 48 physical pins, but it supports dozens of peripherals: multiple serial ports (UART), SPI buses, I²C buses, timers, ADC channels, and more. There aren't enough pins for every peripheral to have dedicated connections, so pins are multiplexed.

Each GPIO pin can be switched from general-purpose mode into an alternate function mode, where it's internally routed to a specific peripheral. For example, pin PA9 could be a regular GPIO, or the transmit line for UART1, depending on how you configure the MODER register.

This flexibility means that part of designing any embedded project is working out the pin assignment before you write code. You decide which peripherals you need, check the datasheet to see which pins support those alternate functions, and plan your circuit board accordingly. Some pins only support specific functions, so this puzzle-solving is a real part of the design process.

Output Speed and Signal Integrity

We briefly mentioned output speed in Part 00. This is where embedded programming starts to overlap with electrical engineering.

When a GPIO pin transitions from low to high, the voltage doesn't snap instantly from 0V to 3.3V. It ramps up over a period of nanoseconds, following a curve determined by the pin's drive circuitry and the electrical characteristics of whatever is connected to it. The slew rate controls how fast this ramp is.

A faster slew rate means the signal reaches its target voltage more quickly, which is essential for high-speed communication protocols like SPI at 10MHz or above. But faster edges also radiate more electromagnetic interference (EMI) and can cause ringing on long wires, where the signal overshoots and oscillates before settling. This ringing can cause false readings on the receiving end.

The rule of thumb: use the slowest speed that still meets your timing requirements. Blinking an LED? Use low speed. Driving an SPI bus at 18MHz? Use very high speed. Getting this right is often the difference between a prototype that works on your bench and a product that works in the field.

Interrupts Revisited: Edge Sensitivity and Priority

Part 00 introduced interrupts as "event listeners for hardware." Here are the details that matter when you actually use them.

Edge sensitivity. You can configure a GPIO interrupt to fire on a rising edge (0→1 transition), a falling edge (1→0), or both. For a button with a pull-up resistor, a falling edge interrupt makes sense because pressing the button pulls the pin from high to low. For a rotary encoder, you'd use both edges to detect direction and speed of rotation.

Interrupt priority. When multiple interrupts fire at the same time (or one fires while another is being handled), the CPU needs to decide which one to service first. ARM Cortex-M microcontrollers have a sophisticated Nested Vectored Interrupt Controller (NVIC) that assigns priority levels to each interrupt source. A motor control interrupt that needs to fire every 50 microseconds would get a higher priority than a button press handler.

Keep it short. Interrupt handlers should do the minimum amount of work necessary: set a flag, copy a value, toggle a pin. Long-running work in an interrupt handler delays other interrupts and can cause the system to miss time-critical events. The pattern is to do the urgent work in the handler and defer everything else to the main loop.
