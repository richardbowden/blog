Timers and PWM

How a microcontroller generates precise timing and waveforms from a single hardware counter, with the CPU free to do other work.

Why Timers Exist

Back in Part 00 we wrote the simplest possible "blink an LED" program: flip a pin high, call delay(500), flip it low, loop. It works, but the CPU is pinned the entire time. It's literally doing nothing except counting clock cycles between the two writes. Every other job the chip could be doing (reading a sensor, servicing a radio, refreshing a screen) is blocked for the full half-second.

Part 01 introduced interrupts as the fix for busy-waiting on external events: button presses, sensor edges, incoming bytes. But plenty of events aren't external. "Toggle this LED every 500ms." "Sample this microphone at exactly 48kHz." "Update this motor controller every 50µs." The event you need to react to is just time passing. You can't attach an interrupt to the wall clock. Microcontrollers solve this with a dedicated piece of hardware that manufactures time-based events for you, called the timer peripheral.

Web dev analogy

In the browser, setInterval(callback, 500) asks the JavaScript runtime to fire your callback every half-second. A microcontroller timer is the same idea, except the runtime scheduling it isn't an event loop sharing the CPU with your code. It's a dedicated counter in silicon that keeps ticking even when the CPU is asleep.

What a Timer Actually Is

A timer peripheral is a hardware counter that increments on every tick of a clock. Three registers control the behaviour:

  • Prescaler (PSC) divides the incoming clock down. If the counter is fed by the 72MHz bus clock on an STM32F103 and PSC = 7199, the counter increments at 72,000,000 / (7199 + 1) = 10,000 Hz, once every 100µs.
  • Counter (CNT) is the running value. It counts up from 0 on every prescaled tick. On a 16-bit timer it can hold values from 0 to 65,535.
  • Auto-reload (ARR) is the ceiling. When CNT reaches ARR, it wraps back to 0 and raises an "update" event. With ARR = 9999, the counter overflows every 10,000 ticks, which at our 100µs tick rate is exactly once per second.

The update event is where a timer earns its keep. It can raise an interrupt, start a DMA transfer, trigger an ADC conversion, or toggle a pin. All of this happens in hardware at precise intervals, whether the CPU is awake, asleep, or busy servicing a different interrupt.

Web dev analogy

A timer is a programmable metronome. You set how often it ticks and what it does on each tick, and the hardware keeps time whether your code is paying attention or not. Reading the counter is a memory read, the same *(volatile uint32_t *)address pattern from Part 01. The counter register is memory-mapped hardware.

Here's the configuration for a 1Hz update event on TIM2, one of the general-purpose timers on an STM32F103:

// TIM2 fed by 72MHz bus clock
TIM2->PSC = 7199;              // 72MHz / 7200  = 10kHz tick rate
TIM2->ARR = 9999;              // overflow every 10,000 ticks = 1Hz
TIM2->DIER |= TIM_DIER_UIE;    // enable update interrupt
TIM2->CR1  |= TIM_CR1_CEN;     // start the counter

void TIM2_IRQHandler(void) {
    TIM2->SR &= ~TIM_SR_UIF;   // clear the update flag
    GPIOA->ODR ^= (1 << 5);    // toggle pin 5
}

Four register writes, one interrupt handler. After that, the CPU is free. It can run your main loop, service other interrupts, or sleep. The timer handles the schedule.

PWM: Pretending to Dim with On/Off

A GPIO pin only knows two states, high or low. So when you see an "LED at 30% brightness" running off a microcontroller, the pin isn't producing an intermediate voltage. It's flipping between fully on and fully off, fast enough that your eye averages the two states into a single perceived brightness. This trick is called pulse-width modulation, or PWM.

PWM has two parameters that matter. The first is duty cycle, the percentage of each period the pin spends high. 0% is always off. 100% is always on. At 50%, the pin spends equal time high and low. The second is frequency, how often the cycle repeats.

Web dev analogy

CSS makes a similar deal. opacity: 0.3 doesn't actually paint pixels at 30% strength. The browser composites the layer over the background, blending pixel values to fake a partially-transparent appearance. PWM is the hardware version of the same trick. The output is always either fully present or fully absent, and the average is what your eye sees.

Same 50% duty cycle, three different frequencies. At 5Hz it's a blink. At 15Hz it's a strobe. At 60Hz most eyes stop perceiving the pulses and you see a steady half-bright LED.

Real PWM rates for LED dimming sit a few hundred hertz to a few kilohertz, well above any visible flicker. Motor controllers run kilohertz-range PWM both to be invisible to the eye and to be quieter than audio frequencies. Audio synthesis pushes into the tens of kilohertz so the carrier itself is inaudible. Only the duty-cycle envelope you're modulating is heard.

Setting up PWM on an STM32 builds on the timer config from the last section. You configure a timer channel for PWM output, and you write the duty cycle into a capture/compare register.

// TIM2 channel 1 -> PWM on PA0
TIM2->PSC    = 71;                  // 72MHz / 72 = 1MHz tick
TIM2->ARR    = 999;                 // overflow every 1000 ticks = 1kHz PWM
TIM2->CCR1   = 500;                 // duty cycle: 500 / 1000 = 50%
TIM2->CCMR1 |= (6 << 4);            // channel 1 in PWM mode 1
TIM2->CCER  |= TIM_CCER_CC1E;       // enable channel 1 output
TIM2->CR1   |= TIM_CR1_CEN;         // start the counter

Six register writes for a continuous 1kHz PWM signal at 50% duty on PA0. Want to dim the LED smoothly? Update CCR1 in your main loop. The hardware reflects the new duty cycle on the very next overflow, with no further help from the CPU.

The same effect on an Arduino is one line:

analogWrite(9, 128);   // pin 9, ~50% duty cycle (range 0-255)

It's the same hardware. Arduino's analogWrite walks through that register configuration in a setup function the first time you call it, then writes a value into the equivalent of CCR1. The bare-metal version shows the configuration. The Arduino version hides it behind a one-line wrapper.
